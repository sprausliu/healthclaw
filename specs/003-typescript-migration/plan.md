# Implementation Plan: TypeScript Migration & Code Quality Uplift

**Branch**: `003-typescript-migration` | **Date**: 2026-02-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-typescript-migration/spec.md`

## Summary

Migrate the HealthClaw webhook-server from JavaScript to TypeScript with strict mode,
enforcing functional programming patterns, immutable data, Zod runtime validation, and
structured logging. The compiler is tsgo (`@typescript-go/tsgo`). All existing HTTP
endpoints, data formats, and pairing flows remain backwards-compatible.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode), Node.js ≥ 22
**Compiler**: tsgo (`@typescript-go/tsgo`) — `tsgo --noEmit` for type check, `tsgo` for build
**Primary Dependencies**:
- `zod` — runtime schema validation + type inference
- `@typescript-go/tsgo` (dev) — TypeScript compiler
- `@typescript-eslint/eslint-plugin` + `@typescript-eslint/parser` (dev) — linting
- `prettier` (dev) — formatting
- `@types/express`, `@types/better-sqlite3`, `@types/node` (dev) — type stubs
**Storage**: better-sqlite3 (SQLite, unchanged), JSONL append-only log (unchanged)
**Testing**: Node.js `node:test` + `supertest` (unchanged test runner)
**Target Platform**: macOS / Linux server (Node.js process)
**Performance Goals**: No regression — existing throughput and latency unchanged
**Constraints**: Zero breaking changes to HTTP API contracts or on-disk data format
**Scale/Scope**: Single server, ~15 source files to migrate

## Key Design Decisions

### 1. tsgo as compiler
tsgo is a Go-based TypeScript compiler that is API-compatible with tsc but significantly
faster for large codebases. Used for both type checking (`tsgo --noEmit`) and build
(`tsgo`). If a compatibility issue is found during migration, fall back to tsc as a
last resort and document the blocker.

### 2. Branded types via intersection
```typescript
type DeviceToken  = string & { readonly _brand: 'DeviceToken' }
type PairingToken = string & { readonly _brand: 'PairingToken' }
type SyncId       = string & { readonly _brand: 'SyncId' }

const toDeviceToken  = (s: string): DeviceToken  => s as DeviceToken
const toPairingToken = (s: string): PairingToken => s as PairingToken
```
Type assertions are only in the factory functions; the rest of the codebase uses the
branded types without `as`.

### 3. Pure function refactoring pattern
```typescript
// BEFORE (impure - reads global, writes to db internally)
async function ingest(record) { ... db.insert(...) }

// AFTER (pure domain logic + injected side effects)
type IngestDeps = {
  readonly isDuplicate: (id: SyncId) => boolean
  readonly save: (record: HealthRecord) => void
}
const ingest = (record: unknown, deps: IngestDeps): IngestResult => { ... }
```

### 4. Zod validation middleware
```typescript
const validate = <T>(schema: ZodSchema<T>) =>
  (req: Request, res: Response, next: NextFunction): void => {
    const result = schema.safeParse(req.body)
    if (!result.success) {
      res.status(400).json({ errors: result.error.flatten() })
      return
    }
    req.body = result.data   // typed, stripped of extra fields
    next()
  }
```

### 5. Structured logger
```typescript
type LogLevel = 'debug' | 'info' | 'warn' | 'error'
type LogEntry = Readonly<{
  level: LogLevel
  timestamp: string
  message: string
  [key: string]: unknown
}>
const log = (level: LogLevel, message: string, ctx?: Record<string, unknown>): void =>
  process.stdout.write(JSON.stringify({ level, timestamp: new Date().toISOString(),
    message, ...ctx }) + '\n')
```

### 6. Migration order (minimise breakage risk)
Migrate leaf modules first, entry point last:
1. Types + schemas (no deps)
2. Logger (no deps)
3. Storage + dedupe (minimal deps)
4. Persistence layer
5. Middleware
6. Services (domain logic refactored here)
7. Entry point (`index.ts`)
8. Tests
